## Question 43

### Difficulty
Medium

### SQL Concepts Covered
* Scalar Subquery

### Objective
Filter data based on an aggregated metric from the entire table.

### Problem Statement
Find the names and populations of all countries whose population is strictly greater than the *global average population*.

### Tables Used
* `country`

### Hint
You cannot put `AVG(Population)` directly into the `WHERE` clause. You must calculate it inside a subquery.

### Expected Learning Outcome
Mastering the Scalar Subquery (a subquery that returns exactly one row and one column).

### Solution (MySQL)
```sql
SELECT 
    Name, 
    Population 
FROM country
WHERE Population > (
    SELECT AVG(Population) FROM country
);
```

### Explanation
* `(SELECT AVG(Population) FROM country)`: This inner query executes first. It calculates the global average. Let's say the average is 25,000,000.
* The outer query then becomes: `WHERE Population > 25000000`.

### Dry Run
1. Inner query runs: `AVG(Population)` = ~25.4 million.
2. Outer query evaluates 'Canada' (31M) -> 31M > 25.4M. Keep.
3. Outer query evaluates 'Aruba' (103k) -> 103k > 25.4M. Discard.

### SQL Execution Order
1. Subquery FROM
2. Subquery SELECT
3. Outer FROM
4. Outer WHERE
5. Outer SELECT

### Alternative Solution
N/A

### Common Mistakes
* Trying to write `WHERE Population > AVG(Population)`. (Invalid syntax because `WHERE` runs before aggregates are calculated).

### Interview Tip
A subquery in the `WHERE` clause using `=`, `>`, or `<` *must* be a Scalar Subquery. If the inner query returns two rows, the database will crash with the error: "Subquery returns more than 1 row."

### Related Concepts
* Aggregate Filtering

## Question 44

### Difficulty
Hard

### SQL Concepts Covered
* IN (with Subquery)

### Objective
Filter the outer query based on a list generated by an inner query.

### Problem Statement
Find the Name of all countries that speak 'Spanish' as an Official Language.

### Tables Used
* `country`
* `countrylanguage`

### Hint
Use a subquery to find all `CountryCode`s in the language table where Language='Spanish' and IsOfficial='T'. Then use `IN` in the outer query.

### Expected Learning Outcome
Using a Multi-Row Subquery with `IN`.

### Solution (MySQL)
```sql
SELECT 
    Name 
FROM country
WHERE Code IN (
    SELECT CountryCode 
    FROM countrylanguage 
    WHERE Language = 'Spanish' AND IsOfficial = 'T'
);
```

### Explanation
* The inner query returns a vertical list (1 column, many rows) of CountryCodes (e.g., 'ARG', 'ESP', 'MEX').
* The outer query evaluates every country's `Code` to see if it exists in that list.

### Dry Run
1. Inner query runs: Returns ['ARG', 'BOL', 'CHL', 'COL', 'CRI', 'CUB', ...].
2. Outer query looks at 'Afghanistan' ('AFG'). Not in list. Discard.
3. Outer query looks at 'Argentina' ('ARG'). In list. Keep.

### SQL Execution Order
1. Subquery FROM, WHERE, SELECT
2. Outer FROM
3. Outer WHERE
4. Outer SELECT

### Alternative Solution
```sql
SELECT co.Name 
FROM country co
INNER JOIN countrylanguage cl ON co.Code = cl.CountryCode
WHERE cl.Language = 'Spanish' AND cl.IsOfficial = 'T';
```
*Pros/Cons:* The `INNER JOIN` approach is often identical in performance because the MySQL optimizer will rewrite the `IN` subquery into a join under the hood (Semi-Join optimization). However, if there were duplicate rows in the language table, the `JOIN` might return duplicate countries, whereas `IN` guarantees unique outer rows.

### Common Mistakes
* Using `=` instead of `IN`. (`WHERE Code = (SELECT...)`). The query crashes because the subquery returns multiple rows.

### Interview Tip
If you only need columns from the Outer table, an `IN` subquery is extremely safe because it avoids the Fan-Out trap. If you join, you risk duplicating rows.

### Related Concepts
* INNER JOIN vs IN
* Semi-Joins

## Question 45

### Difficulty
Hard

### SQL Concepts Covered
* EXISTS
* Correlated Subquery

### Objective
Check for the existence of related rows efficiently.

### Problem Statement
Find the Name of all countries that have AT LEAST ONE city with a population greater than 5,000,000. Use `EXISTS`.

### Tables Used
* `country`
* `city`

### Hint
An `EXISTS` subquery must be linked (correlated) to the outer query, usually by `CountryCode`.

### Expected Learning Outcome
Mastering Correlated Subqueries and the `EXISTS` keyword.

### Solution (MySQL)
```sql
SELECT 
    Name 
FROM country co
WHERE EXISTS (
    SELECT 1 
    FROM city ci 
    WHERE ci.CountryCode = co.Code 
      AND ci.Population > 5000000
);
```

### Explanation
* `EXISTS`: Returns TRUE if the subquery returns 1 or more rows. Returns FALSE if the subquery returns 0 rows.
* `ci.CountryCode = co.Code`: This is a Correlated Subquery. For *every single row* in the `country` table, the database pauses, injects the `co.Code` into the inner query, runs it, and sees if it returns anything.
* `SELECT 1`: We don't care about the actual data, just if the row exists, so selecting a constant `1` is standard practice.

### Dry Run
1. Outer query picks 'USA'.
2. Inner query runs: `SELECT 1 FROM city WHERE CountryCode = 'USA' AND Pop > 5M`. Finds New York. Returns a row.
3. `EXISTS` evaluates to TRUE. USA is kept.
4. Outer query picks 'Aruba'.
5. Inner query runs: `SELECT 1 FROM city WHERE CountryCode = 'ABW' AND Pop > 5M`. Returns 0 rows.
6. `EXISTS` evaluates to FALSE. Aruba is discarded.

### SQL Execution Order
1. Outer FROM
2. Outer WHERE (For each row, executes Subquery FROM, WHERE, SELECT)
3. Outer SELECT

### Alternative Solution
```sql
SELECT DISTINCT co.Name 
FROM country co
INNER JOIN city ci ON co.Code = ci.CountryCode
WHERE ci.Population > 5000000;
```
*Pros/Cons:* The JOIN approach requires a massive hash join and then an expensive `DISTINCT` sort operation to remove the duplicate country names. The `EXISTS` approach is often much faster because as soon as it finds *one* city > 5M for the USA, it completely stops searching the USA cities and moves to the next country (Short-Circuiting).

### Common Mistakes
* Forgetting to correlate the subquery (e.g., leaving out `ci.CountryCode = co.Code`). This makes the inner query return TRUE for *every* country if *any* city in the world > 5M.

### Interview Tip
`EXISTS` vs `IN` is a very common performance question. Historically, `EXISTS` was vastly superior for large datasets. In MySQL 8.0, the optimizer is smart enough to optimize both equally in most cases, but knowing how to write an `EXISTS` query is mandatory for senior roles.

### Related Concepts
* Short-Circuit Evaluation

## Question 46

### Difficulty
Medium

### SQL Concepts Covered
* Nested Subqueries (Subquery in the FROM clause)
* Derived Tables

### Objective
Query a query.

### Problem Statement
Find the average number of cities per country.

### Tables Used
* `city`

### Hint
First, group by `CountryCode` and `COUNT` the cities (this is your subquery). Then, select the `AVG` of those counts from the subquery.

### Expected Learning Outcome
Using Derived Tables (Subqueries in the FROM clause).

### Solution (MySQL)
```sql
SELECT 
    AVG(CityCount) AS AverageCitiesPerCountry
FROM (
    SELECT CountryCode, COUNT(*) AS CityCount
    FROM city
    GROUP BY CountryCode
) AS DerivedTable;
```

### Explanation
* `(SELECT ... ) AS DerivedTable`: The database executes this entire query and holds the result in memory as a temporary table named `DerivedTable`.
* The outer query then treats `DerivedTable` as a real table and calculates the average on the `CityCount` column.

### Dry Run
1. Inner query executes. Creates temporary table: (USA, 274), (CHN, 363), (GBR, 81)...
2. Outer query reads this temporary table and runs `AVG(274, 363, 81...)`.
3. Returns ~17.5.

### SQL Execution Order
1. Inner FROM, GROUP BY, SELECT
2. Outer FROM (The derived table)
3. Outer SELECT

### Alternative Solution
```sql
-- Using a CTE (Common Table Expression) instead of a derived table:
WITH CityCounts AS (
    SELECT CountryCode, COUNT(*) AS CityCount
    FROM city
    GROUP BY CountryCode
)
SELECT AVG(CityCount) AS AverageCitiesPerCountry 
FROM CityCounts;
```
*Pros/Cons:* CTEs are significantly easier to read, especially when nesting multiple levels deep. Derived tables force you to read inside-out.

### Common Mistakes
* Forgetting to alias the derived table. In MySQL, EVERY subquery in the `FROM` clause MUST have an alias (e.g., `AS DerivedTable`), otherwise it throws a syntax error.

### Interview Tip
While this question uses a derived table, CTEs (using the `WITH` keyword) are the modern standard. If an interviewer asks you to write a complex nested query, use a CTE to show you are up-to-date with modern SQL standards.

### Related Concepts
* CTEs (Common Table Expressions)

## Question 47

### Difficulty
Hard

### SQL Concepts Covered
* Subquery in the SELECT clause (Correlated)

### Objective
Attach aggregated data to individual rows without grouping the main query.

### Problem Statement
List every country's Name, its Population, and a third column showing the Total Global Population of the entire world.

### Tables Used
* `country`

### Hint
You cannot put `SUM(Population)` in the `SELECT` clause next to `Name` without a `GROUP BY` (which would ruin the query). You must use a scalar subquery in the `SELECT` clause.

### Expected Learning Outcome
Injecting global metrics using a SELECT subquery.

### Solution (MySQL)
```sql
SELECT 
    Name, 
    Population,
    (SELECT SUM(Population) FROM country) AS GlobalPopulation
FROM country;
```

### Explanation
* The inner query `(SELECT SUM(Population)...)` evaluates to a single number (e.g., 6,078,749,450).
* The outer query selects the `Name`, `Population`, and hardcodes that single number into the third column for every single row.

### Dry Run
1. Inner query runs once. Returns 6.07 Billion.
2. Outer query selects 'USA', 278M, 6.07B.
3. Outer query selects 'Canada', 31M, 6.07B.

### SQL Execution Order
1. Subquery FROM, SELECT (Often cached by the optimizer)
2. Outer FROM
3. Outer SELECT

### Alternative Solution
```sql
-- Using Window Functions (MySQL 8.0+)
SELECT 
    Name,
    Population,
    SUM(Population) OVER() AS GlobalPopulation
FROM country;
```
*Pros/Cons:* The Window Function (`OVER()`) approach is vastly superior. It is cleaner, faster, and designed specifically for this use case. The subquery approach is the legacy way of doing it (pre-MySQL 8.0).

### Common Mistakes
* `SELECT Name, Population, SUM(Population) FROM country` (Throws ONLY_FULL_GROUP_BY error).

### Interview Tip
If asked to calculate a percentage of total (e.g., "What percentage of the global population does China represent?"), this is how you get the denominator. `Population / (SELECT SUM(Population) FROM country)`.

### Related Concepts
* Window Functions (`OVER()`)
